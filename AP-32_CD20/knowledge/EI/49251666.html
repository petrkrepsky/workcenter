Aplikace

Tato technická dokumentace popisuje návrh a funkčnost aplikace(mikroslužby), která se integruje s Kafka topickem DispositionRight a rozhraním zapisPravaPristupovehoUctuKPfDK  vystavené na WSO2. Aplikace čte zprávy z topiku DispositionRight, provádí mapování podle mapovací tabulky a odesílá data do rozhraní SOAP prostřednictvím rozhraní API GW. Zároveň ošetřuje chybové hlášky, které loguje a odesílá do retry topicku JTB-HQ_INTEGRATION_EBROKER_RETRY_APP.


Topic procesing:


Aplikace bude číst topic DispositionRight, Při nové zprávě v topicu, aplikace předčte zprávu, provede mapování a transformaci a následně odešle do eBrokeru.

Mapování najdeme zde DispositionRightsForKUL - Workaround - API.

API GW:


WSDL služba bude vystavena na WSO2 . Specifikace rozhraní je na stránce DispositionRightsForKUL - Workaround - API.


Error handling:


Aplikace zaznamená všechny chyby, které se vyskytnou během volání SOAP API na WSO2. To zahrnuje časové limity(timeout), 500s a 401s. Logy budou uloženy v centralizovaném úložišti logů pro snadný přístup a analýzu.

V případě vypršení časového limitu (timeout) nebo chyby 500,400 při volání SOAP API aplikace odešle data do Retry Topicu. Aplikace má v sobě zabudovaný scheduler(časovač), který hlídá počet opakovaní a ve stanovený čas(configurovatelná hodnota, dle potřeby DKCZ) se zpráva vytahuje z retry topiku a posílá znovu do eBroker. 

Počet opakovaní chybných voláni bude stanoven na 6. Po 6 neúspěšném volání se chyba uloží do logu.



codemessage Poznámka

401Unauthorized  - the client request has not been completed because it lacks valid authentication credentials for the requested resourceV případě chyby 401 při volání SOAP API aplikace zaznamená chybu a odešle data do retry topicu Kafka.

500Internal error - the server encountered an unexpected condition that prevented it from fulfilling the request.V případě chyby 500 při volání SOAP API aplikace zaznamená chybu a odešle data do retry topicu Kafka.

504Timeout - the server, while acting as a gateway or proxy, did not get a response in time from the upstream server that it needed in order to complete the request.V případě chyby 504(Timeout) při volání SOAP API aplikace zaznamená chybu a odešle data do retry topicu Kafka.

Retry Topic:

Pro účely error-handlingu vznikne nový retry topic JTB-HQ_INTEGRATION_EBROKER_RETRY_APP. Kam aplikace zapisuje zprávy z neúspěšného volání.

Topic bude mít stejné schéma jako DispositionRights - Integrační analýza. 

Logs:


Aplikace uloží logy do STDout i do místního souboru v kontejneru. To umožňuje snadný přístup a analýzu logů a pomáhá řešit případné problémy


Container OCP:


Aplikace bude nasazena jako kontejner na OpenShift Container Platform (OCP). To poskytuje bezpečné a škálovatelné prostředí pro nasazení a správu aplikace.

Consumer Group Management:


Aplikace po spuštění naváže na odeslané zprávy podle offsetu své skupiny Consumer v Kafka.  To zajišťuje, že aplikace může sledovat zprávy, které zpracovala, a pokračovat od místa, kde skončila, pokud nastane neočekáváný restart.

Start with Reset Offset:

Aplikace podporuje začátek s resetovaným posunem na začátek Topicku. To umožňuje aplikaci v případě potřeby začít zpracovávat zprávy od začátku Topicku.
