Implemetace

Řešení je postaveno na orchestrační platforme ESB Metada, která orchestruje API volání dvou systému:

1) ASB Sezam - zrušení vazby disponenta

2) CRM - zrušení vazby disponenta

Tato část dokumentace se zabývá implementaci error handlingu řešený pomocí BPMN engine Camunda.

Camunda je open-source platforma pro řízení podnikových procesů a automatizaci pracovních postupů. Poskytuje engine BPMN, který umožňuje modelovat, spouštět a monitorovat obchodní procesy.

Pro spuštění procesu error handling v Camundě je nutné zavolat POST metodu. Volaní bude ale pouze v případě chybné odpovědi ze systému CRM/ASB Sezam. 

V případě, že jeden z těchto systému(případně oba) vrátí chybnou odpověď. ESB Metada inicializuje volání na Camundu pomocí API /process-definition/key/{key}/start, který spouští error handling proces v Camundě. 

Camunda následovně propisuje stavy do topicu ApplicationStateNotification - Integrační analýza, který je odebírán DKCZ/SK.

Metoda

Stávající metoda je součástí API Camunda a spustí process v BPM Camundě. API je vystaveno na WSO2 /process-definition/key/{key}/start. Do paramentru {key} vkládáme hodnotu "cancel-client-disponent-relation" která referuje k procesu zrušení vazby client/disponent.

Zapisování stavů do topicu ApplicationStateNotification

Camunda zapisuje jednotlivé stavy do topicu ApplicationStateNotification. Zapsáni proběhne při změně stavu žádosti. Více infromací o topicu a zápisu notifikaci najdete v integrační analýze ApplicationStateNotification.


Camunda zapisuje do topicu ApplicationStateNotification pomocí X-Correlation-ID, které dostala na vstupu.
