Specifikace požadavků

  • Úprava DDM
  • Zrušení DDM

Sjednocení názvosloví, názvů atributů, výčtových hodnot s topicem

Na rozhraní pro zadávání DDM je potřeba nahradit stavy, které se vrací v odpovědi na zjištění stavu DDM, za business stavy DDM, které jsou uvedené v číselníku odkázaném v DDM - Event Stream - kapitola Popis publikace událostí.

Atribut pro business stav DDM je potřeba pojmenovat tak, aby název reflektoval, že se tímto atributem vrací business stav, nikoliv interní PTS stav.

Je potřeba sjednotit názvy atributů s názvy v topicu, popř. udělat porovnání s REST API na platby a trvalé příkazy, aby všechna rozhraní byla konzistentní, co se týče názvů atributů.

Požadavky na standardizaci

Ošetření chybových stavů a error handling

  • Je potřeba implementovat ošetření chybových stavů podle kapitoly Ošetření chybových stavů v Připojovací analýze.
  • Error handling odpovídá standardu J&T, viz Metodika a standardy J&T. Struktura chybové zprávy by měla odpovídat verzi 2.

Location

  • Požadavkem bezpečnosti je zabránit, aby konzument mohl získat data určená pro jiného konzumenta (CZ a SK digitální kanály).
  • Jednotlivá API je potřeba rozdělit na samostatná API pro CZ a SK se stejnou OpenAPI definicí. Tato API budou rozlišena pomocí přípony CZ a SK v názvu.

Rozšíření request a response parametrů

Na rozhraní pro zadávání DDM je potřeba upravit request a response parametry jednotlivých služeb, aby odpovídaly požadavkům uvedeným v Připojovací analýze - kapitola Synchronní online požadavek a v metodice pro REST API.

Ošetření duplicitních volání - X-Correlation-ID

    • Specifikace požadavku:

      • X-Correlation-ID je 36 znaků plněné UUID.
      • Tento parametr se musí přidat do header parametru na request i response v operacích na vytvoření, změnu či smazání datové entity.
      • Parametr je povinný.
      • Hodnota X-Correlation-ID je v response stejná jako v requestu.
      • Klient (digitální kanály) vyplňuje při volání http header parametr X-Correlation-ID.
      • Primární systém je povinen si při úspěšném zpracování ukládat X-Correlation-ID požadavku pro vytvoření, změnu či mazání datové entity.
      • Pokud v reakci na chybu (timeout, 5xx) provede klient opakování zaslání požadavku, vyplňuje stejné X-Correlation-ID.
      • Na příjem opětovného požadavku se stejným X-Correlation-ID reaguje primární sytém vrácením stejné odpovědi jako u prvotního volání, tj. např. HTTP kód 200 + JSON objekt.
      • Požadavek na uchování X-Correlation-ID pro účel ošetření duplicity je min. 7 dní.
      • Detekce duplicity se týká přijetí požadavku, nikoliv jeho vyřízení.
     X-Correlation-ID: 
      description: "Correlation identifier for request and response pairing."
      required: true
      schema: 
        type: "string"
        format: "uuid"
      example: "4adde395-ebef-4552-bd39-cbdaf63305a9"
Přidání header parametru traceparent

    • Specifikace požadavku:

      • Na rozhraní je požadováno přidat header parametr traceparent pro request i response.
      • Parametr je povinný typu string.
      • Parametr bude vyplňovat konzument rozhraní a bude se propagovat skrze všechny systémy, kterými volání prochází. Momentálně se neřeší jeho napojení do centrálního logování.
      • Hodnota traceparent v requestu je stejná jako v response.
      • Dokumentace: https://www.w3.org/TR/trace-context/
    traceparent: 
      description: "Tracing parameter according to W3C Trace Context standard."
      required: true
      schema: 
        maxLength: 55
        type: "string"
      example: "00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01"
Validace BCD při online požadavku na změnu datové entity

    • Specifikace požadavku:

      • BCD = businessChangeDate = business datum změny, které je svázané s datovou entitou.
      • Jedná se o validaci, zda požadavek na změnu datové entity vychází z posledních informací.
      • Je potřeba přidat nepovinný query parametr BCD na rozhraní.
      • Při příjmu požadavku na změnu datové entity se provede porovnání BCD z požadavku s BCD vedeného u datové entity.
      • Pokud je u datové entity vedena novější hodnota, vrací se chyba s HTTP kódem 428 - Precondition Required.

Definice rozhraní

Changelog - přidání do OpenAPI definice
    • Specifikace požadavku:
      • Při změně rozhraní do OpenAPI definice vložit vždy nový řádek s: datum, verze, autor, seznam změn.
      • Nedělat zpětně.
      • Pokud je OpenAPI definice generovaná a přidání changelogu nelze provést automaticky, je potřeba ho přidat ručně. Rozhraní by měla být dodávána s patřičnou dokumentací - ideálně changelog a manuál.
      • Příklad: # 2022-06-15 – 1.12.0 - Pavel Hrdlička - Added resource "/client", added query parameter "city" in "GET /userInfo"
BID jako path parametr
    • Specifikace požadavku:
      • BID = business ID datové entity
      • V operacích na změnu nebo smazání datové entity je potřeba BID datové entity vložit do path parametru.
Verzování - přidání do OpenAPI definice
    • Specifikace požadavku:
      • Do OpenAPI definice rozhraní je potřeba přidat verzi API.
      • Verze API by měla mít formát MAJOR.MINOR.PATCH, přičemž MINOR a PATCH není povinné (například 1.2.1 nebo 1.1).
      • Nová MAJOR verze se vydá jen v případě zpětně nekompatibilních změn (odstranění resource, přidání povinného parametru...).
      • Změny, které mají pouze informativní charakter (aktualizace kontaktů, doplnění OpenAPI definice, doplnění dokumentace), znamenají navýšení PATCH verze.
      • Verze API na WSO2 se řídí podle verze uvedené v OpenAPI definici rozhraní od BE systému.
      • Odkaz do metodiky: https://confluencedigi.jtfg.com/display/EI/REST+API

Návrh od Credeexperto

Credeexperto již poslalo návrh rozhraní PTSRESTDDM.json.

Datový model v návrhu rozhraní pro request a response odpovídá zadání v solution designech od DK SK a v komunikaci RE_ DDM z DK.msg. Rozhraní již bylo dříve revidované a připomínkované od DK SK. Potvrzeno od DK SK, že navržená podoba operací pro založení, změnu a zrušení DDM je dostačující a rozhraní lze takto implementovat, viz RE_ PTS - DDM API.msg.

Návrh je přesto potřeba zrevidovat a zapracovat všechny požadavky uvedené výše, hlavně následující:

  • sjednocení názvosloví, názvů atributů, výčtových hodnot s topicem
  • požadavky na standardizaci

V návrhu rozhraní je také operace na hledání CID. Seznam CID byl přesunutý a bude získávaný z CSC, tato operace tedy již není potřeba a lze ji z návrhu rozhraní odstranit. Potvrzeno s DK SK, viz RE_ PTS - DDM API.msg.
